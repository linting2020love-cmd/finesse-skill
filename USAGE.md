# 使用清单 — 没有设计经验也能用好 finesse

这不是给 Skill 看的规则文件（那是 `skills/finesse-ui/SKILL.md`，几千行的内部决策逻辑，你不需要读）。这份是给**你**看的：怎么开口、它会怎么回应、怎么确认方向、怎么改、多页面怎么保持一致、最后怎么检查。从头看一遍，以后照着用就行。

---

## 30 秒版

1. 说清楚**页面类型**（落地页/后台/作品集/电商页）+ **1-2 个感觉词**。
2. 它会先回一句判断（Design Read），**你确认了它才动手写代码**。
3. 不满意别说"重做"，**指着说哪里不满意**（一句话，比如"太素了"），它只改那一块。
4. 要做一整套页面，先说"这些要保持一致"，它会锁一份规范，后面每页照着来。
5. 交付前让它跑一遍 `audit`——只读诊断，不改代码，你确认要不要修。

下面每一条展开讲。

---

## 1. 第一次开口：怎么描述你的需求

### 1.1 先说清页面类型——这是最重要的一句话

finesse 把页面分成三条完全不同的路线，说清楚类型，它才知道该照哪一套逻辑排版：

| 类型 | 是什么 | 例子 | 排版思路 |
|---|---|---|---|
| **brand（品牌/落地页）** | 设计本身就是产品——落地页、品牌官网、发布页、作品集、行业 hero 页 | "给瑜伽工作室做个落地页"、"做一个摄影师的个人作品集"、"新产品发布页" | 追求惊艳和第一印象，一屏一个重点，敢用真正的视觉引擎（3D 粒子/Canvas 动效/GSAP 滚动叙事） |
| **product（后台/产品界面）** | 设计是为了服务产品——仪表台、后台、数据分析、设置页 | "给团队做一个数据仪表台"、"电商后台的订单管理页"、"SaaS 的 admin 面板" | 追求清晰和信息密度，不炫技，图表/表格/表单要经得起天天用 |
| **commerce（电商页）** | 卖东西的页面——商品详情页、商品列表、购物车、结算 | "一件旗舰产品的详情页"、"带筛选的商品列表页"、"结算流程" | 卖单品像 brand（有调性），列表页/筛选页像 product（密度高、效率优先） |

**不确定属于哪一类？直接说你不确定**，比如"这个我也说不好是偏展示型还是偏工具型"，finesse 会反问一句帮你判断，不会瞎猜。

### 1.2 感觉词：给 1-2 个最关键的，别堆一串

finesse 内部会把你的形容词换算成三个"旋钮"——**SOUL**（风格辨识度）、**SPECTACLE**（视觉炫技程度）、**DENSITY**（信息密度）。词越精准，换算越准；词堆得越多，反而互相打架。常见词的对应关系大概是这样（不用记，知道有这么回事就行）：

| 你说的词 | 大概会被理解成 |
|---|---|
| "高端"、"奢华"、"高级感" | 风格突出、动效适中、信息不密集——走精品调性 |
| "简约"、"干净"、"克制" | 风格淡一点、动效少、留白多 |
| "大胆"、"冲击力"、"震撼" | 风格和动效都拉满 |
| "编辑感"、"杂志感" | 排版讲究、有动效但不喧宾夺主、内容量适中偏多 |
| "科技感"、"AI 感"、"SaaS 风" | 风格适中、动效较强、信息量中等 |
| "商务"、"B2B"、"企业级" | 风格克制、几乎不炫技、信息量较大 |
| "活泼"、"有创意"、"好玩" | 风格突出、动效中等偏强、内容适中 |
| "数据密集"、"仪表台"、"后台分析" | 几乎不炫技，信息密度拉满 |

- ✅ 好的说法："安静高级"、"有冲击力"、"克制的编辑感"、"暖调复古"
- ❌ 不好的说法："简约又要有科技感又要有温度还要显得高端大气又不能太花哨"（词一多，finesse 也不知道该听谁的，容易做出一个四不像）

如果你举了一个参考（"像 XX 品牌那种感觉"），直接说品牌名或者描述你记得的具体画面，比堆形容词管用得多。

## 2. 它会先给你一句判断——Design Read

finesse 动手写代码之前，**永远**先输出一句判断，格式大概是：

```
Design Read: 深空天文 · 电影感 + 敬畏 · register=brand · SPECTACLE=8 · hero-engine=Three.js 粒子星系
```

拆开看：**行业定位 · 风格方向 · 走的是哪条路线（brand/product/commerce）· 炫技程度打几分 · 用什么技术做视觉核心**。

**这一步会停下来，等你确认。** 你有三种回法：

1. **"可以，继续"/ "go ahead"** —— 它才会真正开始写代码。
2. **指出哪里不对** —— 比如"太科技感了，我们想要更温暖一点"、"这个行业不是这样的调性"。它会重新判断，再给你一句新的 Design Read，不会揣着错误的方向硬写一整页再让你发现全错。
3. **一次性给更多信息** —— 如果它判断的方向大体没错，但漏了什么（比如没考虑到"这个页面主要给老年用户看"），直接补充,它会把这条也纳入判断。

**为什么这一步很重要**：如果跳过确认直接说"随便，你决定"，等做完发现方向不对，返工成本比现在多花 10 秒确认要高得多。

## 3. 拿到第一版之后：怎么反馈、怎么改

### 3.1 核心原则——指着说，别要求整页重做

finesse 有一套针对性调整的命令，你的话会被理解成其中一种，**只改对应的那一块，不会推倒重来**：

| 你可以这样说 | 对应命令 | 会发生什么 |
|---|---|---|
| "太素了" / "想要更有冲击力" / "太无聊了" | `bolder` | 拉高炫技程度（SPECTACLE +2），必要时换一个更重的视觉引擎（比如从 Canvas 换成 Three.js） |
| "太花哨了" / "太吵了" / "感觉很累" | `quieter` | 降低炫技程度（SPECTACLE −2），换成更轻的动效或纯 CSS |
| "感觉不对" / "太千篇一律了" / "不像我们的调性" | `soul` | 换一个风格方向重新来 |
| "太空了" / "内容太少" | `densify` | 提高信息密度，加内容 |
| "太挤了" / "太乱了" | `densify`（反向） | 降低密度，删减内容、加大间距 |
| "加一个 3D 效果" / "想要悬停有质感" | `depth` | 加一个 3D 交互时刻（卡片倾斜/翻转/景深视差），不会大改整体结构 |
| "想换个动画" / "这个动效不好看" | `animate` | 换一个视觉引擎的类型，其余不动 |
| "这块不好看"（具体到某个按钮/配色/组件/文案） | —— | 直接改你点名的那一块，其余原样保留 |
| "这个页面到底好不好，帮我看看" | `audit` | 只诊断、列问题，**不改代码** |
| "这个页面要升级一下" | `redesign` | 会先诊断再改，不是无脑重做整页 |

### 3.2 怎么把反馈说清楚（一个小技巧）

反馈越具体，改得越准：

- ❌ "感觉不太好" —— finesse 不知道从哪下手
- ✅ "hero 区的标题太小了，想要更有气势" —— 明确到区域 + 明确到问题
- ✅ "整体可以，但按钮颜色和页面不搭" —— 明确到组件
- ✅ "移动端上这块挤在一起了" —— 明确到场景

### 3.3 一次只提一类问题，别攒很多轮再一起说

比如同时想要"更有冲击力"又想要"信息更密"，可以一次说清楚，但如果是等做完一版又攒了 5 条新想法，建议先确认这一版方向没问题，再逐条说——不然容易顾此失彼。

## 4. 要做一整套页面（多页项目）：怎么保持"像一家公司出的"

### 4.1 什么时候需要"锁"

如果你只要一个一次性的单页，跳过这一节，直接第 1-3 步就够了。

但如果你要做**官网 + 后台 + 落地页**这种一整套，或者是**长期合作的项目**（这周做官网，下周做产品页，下个月还要加新页面），一定要先说一句：

> "这几个页面要保持一致的风格" / "帮我先建一个项目规范，之后的页面都照这个来"

### 4.2 说了之后会发生什么

finesse 会写一份 `PRODUCT.md`（项目的长期记忆——行业、用户、风格方向、锁定的强度参数、绝对不能像什么）和/或 `design-model.yaml`（更具体的配色/字体/圆角/间距规范）。**这些文件不需要你自己写或维护**，finesse 会自动生成和更新，你只要知道：

- 第一页做完之后，颜色、字体、组件长相都会被记下来。
- 后面每一页开工前，finesse 会先读这份记忆，照着做，不会自己发挥换配色换字体。
- 如果你后面某一页确实想要不一样的调性（比如后台想比官网更克制），直接说清楚原因，finesse 会记录"这一页为什么例外"，而不是悄悄破坏整体一致性。

### 4.3 已经有代码库了怎么办

如果不是从零开始，项目里已经有别人做的页面，可以说"帮我看看现在这套页面用的是什么设计系统"——finesse 会读现有代码，反推出一份 `design-model.yaml`，之后新页面就会照着这套已有风格走，而不是自己另起一套。

---

## 5. 交付前最后一步：自查

说一句"帮我检查一下这个页面"或者直接说 `audit`，finesse 会做一次**只读诊断**（不改任何代码），对照几类硬性规则打分：

- **方向是否统一**——配色有没有跑偏、有没有莫名其妙冒出来的第二个主题色
- **有没有"一眼假"的 AI 套路**——比如乱用破折号、渐变文字、千篇一律的图标卡片墙、编造的精确数字（"92%"、"4.1×"却没有真实数据支撑）
- **无障碍/可用性**——文字对比度够不够、按钮在手机上点得到点不到、键盘能不能操作、动效是否尊重"减少动态效果"的系统设置
- **性能**——图片/动效会不会拖慢页面、移动端有没有横向滚动条

诊断完只会给你一份问题清单（分优先级），**不会自己动手改**。你看完可以说"先把 XX 改一下"，它才会去改——你始终掌握"要不要改、先改哪个"的决定权。

## 6. 三个完整例子

### 例子一：品牌落地页（brand）

```
你：给一个精品咖啡烘焙工坊做落地页，要暖调、有点复古的质感

finesse：Design Read: 精品咖啡 · 暖调复古工艺 · register=brand · SPECTACLE=5 · hero-engine=Canvas 粒子/火焰
         （停下等你确认）

你：可以，继续

finesse：（生成完整页面）

你：整体不错，但 hero 区太素了，想要更有冲击力一点

finesse：（只调整 hero 区的炫技程度，其余不动）

你：帮我看看这页现在怎么样

finesse：audit 只读诊断：✅ 反廉价黑名单通过 ⚠️ 移动端触摸目标偏小 ⚠️ 一处对比度不足

你：先把触摸目标改一下

finesse：（只改这一处）
```

### 例子二：产品后台（product）

```
你：给运维团队做一个基础设施监控仪表台，深色主题

finesse：Design Read: 基础设施监控 · 冷静精确 · register=product · SPECTACLE=2 · component-system
         （停下等你确认）

你：可以

finesse：（生成侧边栏+图表+数据表的完整仪表台）

你：图表区太挤了，看着头晕

finesse：densify（反向）—— 精简卡片数量、拉开间距，突出最关键的 2-3 个指标

你：这几个页面（监控台+告警页+设置页）以后要保持一致

finesse：（建立 design-model，记录配色/字体/组件规范，之后每页照此生成）
```

### 例子三：电商详情页（commerce）

```
你：一款旗舰耳机的商品详情页，希望有质感

finesse：Design Read: 高端耳机 · 精密工程感 · register=commerce(PDP,偏brand) · SPECTACLE=6 · hero-engine=产品摄影+轻量视差
         （停下等你确认）

你：可以，继续

finesse：（生成详情页：hero大图+规格+评价+购买区）

你：购买按钮不够醒目

finesse：（只调整 CTA 按钮的视觉权重，其余不动）
```

## 7. 常见问题 FAQ

**Q：我完全不会说设计术语，只会说"好看/不好看"，可以吗？**
可以，但越具体越好。"不好看"可以换成"感觉太老气了"或者"太花哨了"这种更有方向感的词，finesse 能理解的信息越多，第一版就越准，你要返工的次数就越少。

**Q：它生成的东西我一个字都不会改代码，出问题了怎么办？**
直接把问题描述给它就行（"这块在手机上显示不全"），不需要你懂代码，finesse 会自己定位问题并修。

**Q：一定要先确认 Design Read 吗，能不能让它直接一次做完？**
可以跳过，直接说"随便你，做就行"，但不推荐——这一步只要几秒钟，能避免方向完全错了之后的大返工。

**Q：多个页面之间风格跑偏了怎么办，还能救吗？**
可以。说"帮我看看这几个页面风格统一吗"，finesse 会对比现有页面反推一份规范，再挨个调整跑偏的地方，不需要全部推倒重做。

**Q：我应该用哪个工具（Claude Code / Cursor / Codex / Copilot）？**
都支持，装法看主 README 的"安装 & 工具支持"一节，用法（也就是这份文档讲的内容）在所有工具里都是一样的。
